# Bookmark Sync（Chrome / Edge）

将浏览器书签同步到你自己的 GitHub / Gitee 仓库。  
适用于多设备同步、私有化备份和手动可控的恢复场景。

## 项目说明

插件会把本地书签导出成快照 JSON，并读写到仓库指定路径（默认 `bookmarks/snapshot.json`）。

核心动作：

- `推送本地到云端`：上传本地书签到仓库
- `拉取云端覆盖本地`：下载仓库快照并覆盖本地书签
- `导出到 JSON 文件`：本地离线备份
- `从 JSON 文件导入`：从离线文件恢复

---

## 功能与边界

已支持：

- GitHub / Gitee 双平台
- 自动定时推送（最小 15 分钟）
- 同步冲突检测与差异预览
- 手动强制覆盖（冲突后可选）

当前边界：

- 自动任务只执行“推送”，不会自动拉取
- 冲突时自动任务会跳过，不会强制覆盖
- 拉取属于覆盖导入操作，建议先备份

## 安装插件

### Chrome

1. 打开 `chrome://extensions`
2. 开启“开发者模式”
3. 点击“加载已解压的扩展程序”
4. 选择本项目根目录（含 `manifest.json`）

### Edge

1. 打开 `edge://extensions`
2. 开启“开发人员模式”
3. 点击“加载解压缩的扩展”
4. 选择本项目根目录（含 `manifest.json`）

## 配置项说明

在“打开设置”页面填写：

| 字段 | 说明 | 示例 |
|---|---|---|
| 同步提供方 | `GitHub` 或 `Gitee` | `GitHub` |
| Token | 平台访问令牌 | `ghp_xxx` |
| 用户/组织 | owner 名称 | `your-name` |
| 仓库名 | repo 名称 | `bookmark-sync` |
| 分支 | 目标分支 | `main` / `master` |
| 文件路径 | 快照文件路径 | `bookmarks/snapshot.json` |
| 启用自动推送 | 是否开启定时推送 | 开 / 关 |
| 自动推送间隔(分钟) | 最小 15 分钟 | `60` |

建议默认：

- 文件路径使用 `bookmarks/snapshot.json`
- 先手动同步跑通，再开启自动推送

## GitHub 配置流程

### 1. 创建仓库

1. 在 GitHub 新建仓库（建议 `Private`）
2. 记录：
   - 用户名/组织名（owner）
   - 仓库名（repo）
   - 默认分支（通常 `main`）

### 2. 创建 Token

路径：  
`Settings -> Developer settings -> Personal access tokens -> Fine-grained tokens`

关键设置：

1. `Repository access` 选择 `Only select repositories`
2. 选择目标仓库
3. 权限 `Contents` 设为 `Read and write`
4. 生成后复制 Token（只显示一次）

### 3. 填写插件设置

1. 插件弹窗点击 `打开设置`
2. 填写：
   - 同步提供方：`GitHub`
   - Token：上一步生成的 Token
   - 用户/组织：owner
   - 仓库名：repo
   - 分支：`main`（按实际）
   - 文件路径：`bookmarks/snapshot.json`
3. 点击 `保存设置`
4. 点击 `测试连接`

测试通过后即可同步。

## Gitee 配置流程

### 1. 创建仓库

1. 在 Gitee 新建仓库（建议私有）
2. 记录：
   - 用户名/组织名（owner）
   - 仓库名（repo）
   - 默认分支（常见 `master`，也可能 `main`）

### 2. 创建私人令牌

在 Gitee 个人设置中创建“私人令牌（Personal Access Token）”，权限需支持仓库内容读写。

### 3. 填写插件设置

1. 插件弹窗点击 `打开设置`
2. 填写：
   - 同步提供方：`Gitee`
   - Token：私人令牌
   - 用户/组织：owner
   - 仓库名：repo
   - 分支：按实际分支填写
   - 文件路径：`bookmarks/snapshot.json`
3. 点击 `保存设置`
4. 点击 `测试连接`

测试通过后即可同步。

## 首次同步策略

首次操作建议先确定“以哪一边为准”：

- 以当前本地书签为准：执行 `推送本地到云端`
- 以仓库快照为准：执行 `拉取云端覆盖本地`

如果远端尚无快照文件，请先执行一次“推送”。

## 日常使用流程

### 单设备

书签有改动后，执行一次 `推送本地到云端` 即可。

### 多设备

推荐顺序：

1. 在设备 A 修改后先推送
2. 在设备 B 先拉取再修改
3. 修改完成后立即推送

该顺序可显著降低冲突概率。

---

## 自动同步与冲突处理

### 自动同步

在设置页开启 `启用自动推送` 并设置间隔（>=15 分钟）。

注意：

- 自动同步只执行推送
- 冲突时会跳过，不执行强制覆盖

### 冲突处理

检测到冲突时，插件会给出差异预览（本地与远端统计 + 样例条目），你可以：

1. 取消本次同步
2. 选择强制继续（会覆盖另一侧）

建议先导出本地 JSON 备份，再执行强制操作。

## 备份与恢复

- 备份：点击 `导出到 JSON 文件`
- 恢复：点击 `从 JSON 文件导入`

这是最直接的兜底手段，建议在大范围变更前执行一次备份。

## 常见问题

| 现象 | 原因 | 处理方式 |
|---|---|---|
| 缺少配置项 | 有必填字段为空 | 检查 Token / owner / repo / branch / path |
| 远端文件不存在 | 先拉取但远端尚未创建快照 | 先执行一次推送 |
| 401 / 403 | Token 无效或权限不足 | 重新生成 Token 并确认仓库读写权限 |
| 一直连接失败 | owner/repo/branch 填写错误 | 逐项核对仓库信息 |
| 冲突频繁 | 多设备并发修改 | 执行“先拉取再修改，改完即推送” |

---

## 安全建议

1. 使用私有仓库
2. Token 仅授予必要权限
3. 不在公开场景泄露 Token
4. 定期轮换 Token
5. 冲突覆盖前先做 JSON 备份

3. 首次明确“以本地为准”或“以远端为准”
